purpose

01 / 10

PersonalSite is Adrian Lumley’s public editorial and product-leader surface: work, writing, experiments, an episodic Signal Room, and a direct contact path.

The build is primarily static. Astro turns route files, typed content, shared components, and local assets into deployable output. A few browser-side integrations are deliberately non-critical: analytics, contact submission, and a public status read.

The governing shape is editorial, not operational. Shared systems should make the surface more coherent without making it look like a control panel.

signal room

An episodic collection with explicit identity and index presentation fields.

required
title, episode, date, teaser, badges
identity
episode number is the ordering authority
ordering
ascending or descending through one helper
route
/signal-room/<slug>/

Content bodies are intentionally outside the architecture fingerprint. Schemas, shared queries, and route implementations stay inside it, so structural changes invalidate the map while ordinary publication does not.

data flow

06 / 10
  1. page source

    Route files select content and page-specific components.

  2. shared composition

    Components, navigation, metadata, and Base.astro establish one document system.

  3. Astro build

    Static HTML, inline CSS, endpoint output, assets, feeds, and images are written to dist/.

  4. delivery

    GitHub Pages serves the built artifact; optional browser integrations fail independently.

  1. Markdown body + frontmatter

    Collection schemas reject invalid writing or episode metadata.

  2. canonical queries

    getPublishedPosts and getEpisodes apply shared visibility and ordering.

  3. multiple consumers

    Indexes, entry pages, RSS, SEO, and Open Graph output read the same normalized layer.

  4. stable identifiers

    Central route helpers preserve historical path forms used by feed readers.

protected boundaries

09 / 10
  • public surfacesrc/ and public/ are protected for contribution-only work. The existing src/agent/ gate tests are the narrow documented exception.
  • dependencies + buildManifests, lockfiles, Astro, Vitest, Playwright, and related configuration do not change without explicit product scope.
  • delivery + services.github/, deploy configuration, authentication, worker state, and external services are never incidental edit targets.
  • portable contextContracts and docs use repo-relative paths and exclude credentials, private logistics, and machine-specific details.

escalate when doctrine conflicts would change behavior, verification needs installation or secrets, a boundary cannot be proven from the repository, or an action would touch live state.

agent contribution path

10 / 10

Read for authority first, use the machine map for implementation orientation, then let the gate test both contracts before work leaves the repository.

  1. REVIEW.mdshort onboarding brief and work classification
  2. AGENTS.mdhighest-precedence public-site doctrine and verification requirements
  3. CLAUDE.mdrepository structure, content conventions, and deployment context
  4. docs/zero-context-contribution.mdauthority precedence, boundaries, and contribution procedure
  5. .agent/contribution-contract.jsonenforceable contribution and verification contract
  6. .agent/architecture.jsonmachine-readable implementation map, interfaces, flows, invariants, and source fingerprint
  7. contribution_gate.py auditdeterministic pre-edit and pre-commit architecture/context audit

Read order accelerates onboarding. Authority precedence remains AGENTS.md, CLAUDE.md, REVIEW.md, .agent/contribution-contract.json, then docs/zero-context-contribution.md.
